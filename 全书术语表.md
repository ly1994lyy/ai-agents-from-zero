# 全书术语表

这份术语表不是为了把名词堆得越来越多，而是为了帮你把整套教程里最容易混淆的概念真正理顺。  
它重点解决四类问题：

- 同一个概念在不同章节、不同框架、不同平台里换了名字，不敢确认是不是同一回事
- 概念都见过，但说不清它们分别解决什么问题
- 会照着案例写代码，却讲不清“为什么这里该用这个，不该用那个”
- 学到后半段以后，前面的 Prompt、RAG、Tool、Workflow、Agent、MCP、LangGraph 开始互相打架

---

## 1、怎么使用这份术语表

- 这份术语表优先采用**本教程内部统一口径**，目标是帮你学整套课时不乱。
- 同一个概念如果在官方资料里常见别名，会在“说明 / 易混点”里补出来。
- 每个术语都尽量给出“建议先看”章节，方便你顺着回到原文。
- 不要求一次性背完。你最应该关注的是：**它解决什么问题、和相邻概念的边界在哪里。**

---

## 2、先抓住整套课的 8 条主线

如果你暂时不想看大表，可以先把这 8 条主线记住：

1. **Prompt** 主要解决“怎么问、怎么约束、怎么把任务说清楚”。
2. **RAG** 主要解决“模型不知道、知识会过期、资料在外部”。
3. **微调** 主要解决“模型行为、风格、格式、偏好不稳定”。
4. **Workflow** 主要解决“流程固定、步骤明确、强调稳定可控”。
5. **Agent** 主要解决“流程不固定，需要模型边做边决定下一步”。
6. **Tool / Function Calling** 主要解决“模型如何表达‘我想调用哪个工具、传什么参数’”。
7. **MCP** 主要解决“外部能力如何被标准化暴露给 AI 应用”。
8. **LangGraph** 主要解决“复杂状态化流程如何编排、持久化、观察和扩展”。

只要你不把这 8 条主线搞混，整套教程的大部分概念边界就不会乱。

| **续训**         | 在已有基座上继续预训练         | 更偏补领域知识分布，不是直接调行为风格               | [1-3](1-3-RAG、微调、续训与智能体选型.md)                                     |
| **微调**         | 用更针对性的训练数据改模型行为 | 更偏格式、风格、任务习惯、偏好                       | [1-3](1-3-RAG、微调、续训与智能体选型.md)                                     |
| **SFT**          | 监督微调                       | 微调里最常见的一种方式，用标注好的输入输出样本训练   | [1-1](1-1-大模型认知与工程概览.md)、[1-3](1-3-RAG、微调、续训与智能体选型.md) |
| **RLHF / RLAIF** | 基于人类 / AI 反馈的对齐训练   | 更偏“让模型更符合偏好和规范”                         | [1-1](1-1-大模型认知与工程概览.md)                                            |
| **LoRA**         | 低秩适配微调                   | 用较低成本对模型做增量训练                           | [1-3](1-3-RAG、微调、续训与智能体选型.md)                                     |
| **QLoRA**        | 量化版 LoRA 微调方案           | 常用于显存更紧张的微调场景                           | [1-3](1-3-RAG、微调、续训与智能体选型.md)                                     |
| **灾难性遗忘**   | 新训练后把原来会的东西忘掉     | 微调和续训都要警惕                                   | [1-3](1-3-RAG、微调、续训与智能体选型.md)                                     |
| **幻觉**         | 模型一本正经地说错话           | RAG 能缓解，但不能保证彻底消除                       | [1-1](1-1-大模型认知与工程概览.md)、[19](19-RAG检索增强生成.md)               |
| **推理**         | 模型对外执行回答和生成的阶段   | 训练结束后，线上调用主要发生在推理阶段               | [1-1](1-1-大模型认知与工程概览.md)、[11](11-Model-I-O与模型接入.md)           |
| **Temperature**  | 控制输出随机性的参数           | 越高通常越发散，越低通常越稳定；不是“越低越好”       | [11](11-Model-I-O与模型接入.md)                                               |
| **Top P**        | 基于概率质量的采样控制参数     | 和 Temperature 都会影响输出多样性                    | [11](11-Model-I-O与模型接入.md)                                               |

| -------------------------- | ------------------------------------ | -------------------------------------------------------------------- | ------------------------------------------------------------------------ |
| **Prompt**                 | 给模型的输入说明                     | 从一句话到复杂模板都属于 Prompt                                      | [1-2](1-2-提示词工程基础.md)、[13](13-提示词与消息模板.md)               |
| **提示词工程**             | 系统设计 Prompt 的方法               | 重点不是“写漂亮话”，而是让任务边界更清晰、结果更稳定                 | [1-2](1-2-提示词工程基础.md)                                             |
| **Zero-shot**              | 不给示例，直接让模型做任务           | 适合简单清晰任务                                                     | [1-2](1-2-提示词工程基础.md)                                             |
| **Few-shot**               | 给少量示例再让模型做任务             | 适合格式、风格、模式需要被“带一下”的任务                             | [1-2](1-2-提示词工程基础.md)                                             |
| **System Message**         | 系统级规则消息                       | 常用于定义角色、边界、风格、硬约束                                   | [1-2](1-2-提示词工程基础.md)、[13](13-提示词与消息模板.md)               |

| **AI / Assistant Message** | 模型消息                             | 既可以是历史回复，也可以是带工具调用痕迹的中间消息                   | [11](11-Model-I-O与模型接入.md)、[13](13-提示词与消息模板.md)            |
| **PromptTemplate**         | 文本模板                             | 适合单段文本 Prompt                                                  | [13](13-提示词与消息模板.md)                                             |
| **ChatPromptTemplate**     | 聊天消息模板                         | 适合多角色、多轮消息结构                                             | [13](13-提示词与消息模板.md)                                             |
| **MessagesPlaceholder**    | 消息占位符                           | 用来把历史消息动态插回 Prompt 里                                     | [13](13-提示词与消息模板.md)、[16](16-记忆与对话历史（含Redis基础）.md)  |
| **Model I/O**              | 模型输入输出层                       | 可以粗理解为“怎么喂给模型、怎么拿回结果、怎么把结果变成程序可用结构” | [11](11-Model-I-O与模型接入.md)                                          |
| **AIMessage**              | LangChain 中的模型回复对象           | 不只是 `.content` 文本，还可能带 metadata、tool_calls 等信息         | [11](11-Model-I-O与模型接入.md)                                          |
| **invoke**                 | 一次性同步调用                       | 最常见的执行方式                                                     | [10](10-LangChain快速上手与HelloWorld.md)、[15](15-LCEL与链式调用.md)    |
| **batch**                  | 批量调用                             | 适合一组输入同时处理                                                 | [15](15-LCEL与链式调用.md)                                               |
| **stream**                 | 流式输出                             | 边生成边返回，更适合实时交互和观测过程                               | [10](10-LangChain快速上手与HelloWorld.md)、[25](25-LangGraph高级特性.md) |
| **Output Parser**          | 输出解析器                           | 把模型原始输出整理成程序更好处理的结构                               | [14](14-输出解析器.md)                                                   |
| **Structured Output**      | 结构化输出                           | 让模型按指定字段或 schema 输出                                       | [14](14-输出解析器.md)、[21](21-Agent智能体.md)                          |
| **TypedDict**              | 轻量结构定义方式                     | 更偏结构声明，不等于强运行时校验                                     | [14](14-输出解析器.md)                                                   |
| **Pydantic**               | Python 数据校验库                    | 常用于 schema 定义和运行时校验                                       | [14](14-输出解析器.md)、[17](17-Tools工具调用.md)                        |
| **JSON Schema**            | JSON 结构约束规范                    | 常用于描述字段类型、必填项、枚举值等                                 | [14](14-输出解析器.md)                                                   |
| **response_format**        | Agent / 模型结果结构约束入口         | 在 LangChain 1.x 语境里经常用于让 Agent 最终输出按结构返回           | [21](21-Agent智能体.md)                                                  |
| **ProviderStrategy**       | 由模型提供方原生支持结构化输出的策略 | 前提是底层模型 / Provider 真支持                                     | [21](21-Agent智能体.md)                                                  |
| **ToolStrategy**           | 借助工具调用机制实现结构化输出的策略 | 常用于没有原生结构化能力时的替代方案                                 | [21](21-Agent智能体.md)                                                  |

| **切块 / Chunking**   | 长文切成小段                         | 切得太大，噪声高；切得太碎，语义不完整             | [19](19-RAG检索增强生成.md)                                            |
| **Chunk Overlap**     | 相邻切块的重叠区域                   | 用来减少切块断裂导致的信息损失                     | [19](19-RAG检索增强生成.md)                                            |
| **元数据 / Metadata** | 附加在文档块上的结构信息             | 比如来源、时间、标题、标签；可用于过滤和解释结果   | [18](18-向量数据库与Embedding实战.md)、[19](19-RAG检索增强生成.md)     |
| **Retriever**         | 检索器                               | 负责从知识库里找候选片段                           | [19](19-RAG检索增强生成.md)                                            |
| **召回**              | 从知识库里先找出一批候选内容         | 重点是“别漏掉该找的”，不等于最终排序最好           | [18](18-向量数据库与Embedding实战.md)、[19](19-RAG检索增强生成.md)     |
| **Rerank / 重排**     | 对召回结果再排序                     | 目的是把真正最有用的片段放前面                     | [8](8-企业级大模型部署.md)、[19](19-RAG检索增强生成.md)                |
| **过滤 / Filter**     | 按元数据条件缩小检索范围             | 常用于租户隔离、时间范围、文档类型限制             | [19](19-RAG检索增强生成.md)                                            |
| **RAG**               | 检索增强生成                         | 先从外部资料中查，再基于查到的资料生成回答         | [1-3](1-3-RAG、微调、续训与智能体选型.md)、[19](19-RAG检索增强生成.md) |
| **索引阶段**          | 准备知识库的阶段                     | 包括加载文档、切块、向量化、入库                   | [19](19-RAG检索增强生成.md)                                            |
| **检索阶段**          | 用户提问时查找候选内容的阶段         | 是运行时行为                                       | [19](19-RAG检索增强生成.md)                                            |
| **生成阶段**          | 把检索结果与问题一起喂给模型生成答案 | 不是简单“把片段拼进去”就完了，还要考虑 Prompt 约束 | [19](19-RAG检索增强生成.md)                                            |
| **管道式 RAG**        | 检索步骤基本固定的 RAG               | 更好调试、更适合先学清基础链路                     | [19](19-RAG检索增强生成.md)                                            |
| **Agent 式 RAG**      | 是否检索、怎么检索交给 Agent 决定    | 灵活度更高，但复杂度和不确定性也更高               | [19](19-RAG检索增强生成.md)、[21](21-Agent智能体.md)                   |

## 12、最容易混淆的概念对照

### 12.1 RAG vs 微调 vs 续训

- **RAG**：补运行时外部知识，不改模型参数
- **微调**：改行为、格式、风格、任务习惯
- **续训**：补更底层的领域知识分布，成本通常更高

一句话判断：**缺资料先想 RAG，缺行为再看微调，底层知识分布都不对才考虑续训。**

### 12.2 Workflow vs Agent

- **Workflow**：流程由人设计，模型只在既定位置完成子任务
- **Agent**：模型参与决定下一步要做什么

一句话判断：**路径固定优先 Workflow，路径开放再考虑 Agent。**

### 12.3 Tool / Function Calling vs MCP

- **Tool / Function Calling**：模型如何表达“我要调用这个工具”
- **MCP**：外部能力如何被标准化暴露出来

一句话判断：**一个偏调用机制，一个偏接入协议。**

### 12.4 MCP vs A2A

- **MCP**：让模型接外部工具、资源、提示模板
- **A2A**：让不同 Agent / Agent 系统互相协作

一句话判断：**一个是接能力，一个是接另一个 Agent。**

### 12.5 LangChain vs LangGraph

- **LangChain**：组件、模型、Prompt、Parser、Tools、Agent 封装层
- **LangGraph**：图式编排、状态流转、持久化、恢复、观察层

一句话判断：**LangChain 更像“拼能力”，LangGraph 更像“跑流程”。**

### 12.6 对话历史 vs 记忆 vs Checkpoint

- **对话历史**：消息本身
- **记忆**：系统利用历史的能力
- **Checkpoint**：某次执行过程中的状态快照

一句话判断：**历史是内容，记忆是能力，checkpoint 是状态保存机制。**
